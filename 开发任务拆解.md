## 2. MVP 闭环

第一条必须跑通的闭环路径：

```
ncatbot mock event (GroupMessage/PrivateMessage)
  → IncomingMessageQueue.put(raw_event)
  → MessageIngestor.drain()
    → RawMessageLog.insert_if_new(msg.message_id)
    → ShortTermMemory.add_incoming()
  → ShortTermMemory.consume_unread()
  → PassiveReplyJudge.evaluate()
    → ActionPlan
  → ContentGenerator.generate()
    → GeneratedAction
  → EmergencyBrake.final_check()
  → DelaySendScheduler.schedule()
    → pre_send_check()
    → BotAdapter.send() [mock]
    → ActionLog.mark_as_sending() + record_send_result()
    → StateEventQueue.put()
  → BotState.apply()
```

### MVP 中哪些模块用 mock

| 模块 | MVP 处理方式 |
|------|------------|
| `BotAdapter` | Mock：send() 返回 `SendResult(status=SendStatus.SENT, ...)` 不真正调 ncatbot |
| `LLMClient` | Mock 或固定返回：ContentGenerator 的 LLM 调用返回固定 JSON |
| `ncatbot callback` | 手动构造 `GroupMessage` / `PrivateMessage` 对象放入 queue |
| `LongTermMemory` | 跳过，MVP 不做浓缩 |
| `ConcentrateJob` | 跳过 |
| `RelationshipGraph` / `EntityProfile` | 跳过 |
| `ActivityCurve` | 硬编码返回 1.0（全时段活跃） |
| `EmotionEngine` | 跳过，固定 valence=0, energy=1, social_battery=1 |
| `ShutdownManager` | 跳过，Ctrl+C 直接退出 |

### MVP 中哪些功能必须真实实现

| 模块 | 理由 |
|------|------|
| `Config` / `ConfigLoader` | 所有模块都依赖配置 |
| `Message` / `ActionPlan` / `GeneratedAction` schema | 数据流转的骨架 |
| `IncomingMessageQueue` | 线程安全入口 |
| `MessageIngestor.drain()` + `_parse()` | 消息解析 |
| `RawMessageLog` | 去重 |
| `ShortTermMemory` | 消息暂存 |
| `PassiveReplyJudge` | 被动回复判断 |
| `DecisionEngine.decide(PASSIVE)` | 决策入口 |
| `ContentGenerator.generate()` | 生成回复 |
| `EmergencyBrake.final_check()` | 内容过滤 |
| `DelaySendScheduler` | 延迟发送 |
| `ActionLog` | 状态追踪 |
| `StateEventQueue` + `BotState.apply()` | 状态传递 |
| `Orchestrator.run_passive_tick()` | 串联一切 |
| `BotState` (最小版) | mode + tick() |

### MVP 闭环成功的判定标准

1. 手动放入一个 mock `@bot` 的 GroupMessage → 3 秒内产生 ActionPlan
2. ActionPlan 经 ContentGenerator 生成 GeneratedAction（含文本内容）
3. GeneratedAction 通过 EmergencyBrake → 进入 DelaySendScheduler
4. DelaySendScheduler 到期后调用 BotAdapter.send()（mock），ActionLog 状态变为 `sent`
5. StateEvent 被 BotState.apply() 消费
6. 全流程无异常、无死锁、ActionLog 中无僵尸 `planned` 记录

## 7. 编码约束

1. **跨线程写状态必须走 StateEventQueue** — BotState 只能由主循环线程通过 `apply()` 修改
2. **BotState 禁止被 DelaySendScheduler 直接修改** — 发送结果通过 StateEventQueue 传递
3. **DecisionEngine 禁止调用 ContentGenerator** — 决策与生成硬隔离
4. **PassiveReplyJudge 禁止调用 LLM** — 纯规则判断
5. **ActiveSpeakGate 只允许调用 LLM 做主动发言意图判断** — 输出仅限 DecisionOutput (should_speak/chat_id/intent/reason)
6. **ContentGenerator 输出必须经过 Pydantic schema 校验** — 不信任 LLM 输出
7. **BotAdapter 不能返回 bool，必须返回 SendResult** — 含 status、error、latency_ms
8. **ActionLog 所有状态变更必须幂等** — mark_as_sending 用 SQL WHERE status IN (...) 保证
9. **RawMessageLog 必须保证 message_id 去重** — UNIQUE 约束 + INSERT OR IGNORE
10. **pre_send_check MVP 阶段禁止调用 LLM** — 只能用纯规则
11. **ShortTermMemory._read/_pending 只允许主循环线程写入** — 通过代码审查保证
12. **SQLite 默认每个组件/线程持有独立 connection，同一 connection 不跨线程共享** — 需要跨线程使用同一 connection 时必须显式加锁；建议开启 WAL 模式
13. **配置文件（bot.yaml, character.yaml, group_name.yaml）不提交 git** — 只提交 .template 版本
14. **LLM 调用必须有超时（30s）和重试（最多 3 次，指数退避）**
15. **LLMClient 只返回原始文本（chat_text）** — JSON 提取和 Pydantic schema 校验由各调用方负责（ContentGenerator / ActiveSpeakGate / ConcentrateJob）

## 8. 风险与降级

### R1：ncatbot 回调线程阻塞

- **风险描述**：ncatbot 事件回调如果执行耗时操作，会阻塞后续事件处理
- **触发条件**：回调中直接做消息解析、去重、写 MemoryManager
- **影响**：消息积压、事件丢失
- **预防措施**：回调线程只做 `queue.put(raw_event)`，零处理
- **测试方式**：mock 1000 条消息快速注入，验证 queue 不溢出、无丢失

### R2：ActionLog 状态不一致

- **风险描述**：action 记录停留在 `planned` 状态，成为僵尸记录
- **触发条件**：LLM 调用失败后未更新 action 状态
- **影响**：action_log 膨胀、调试困难
- **预防措施**：所有 LLM 失败路径必须 `record_batch(status='cancelled')`；定期扫描僵尸记录
- **测试方式**：注入 LLM 失败 → 验证 action_log 无 `planned` 超过 1 分钟

### R3：StateEvent 未及时 apply

- **风险描述**：StateEvent 堆积在 queue 中，BotState 状态滞后
- **触发条件**：主循环阻塞（如 LLM 调用超时 30s）
- **影响**：BotState 情绪/能量不准确，决策基于过时状态
- **预防措施**：每轮 tick 前先 `_drain_state_events()`；LLM 调用有超时
- **测试方式**：快速注入 100 个 StateEvent → 验证下一轮 tick 前全部 apply

### R4：DEGRADED 无法恢复

- **风险描述**：进入 DEGRADED 后永远无法切回 RUNNING
- **触发条件**：health_check 逻辑错误，或 LLM 恢复后未正确 emit LLM_RECOVERED
- **影响**：bot 永久使用模板回复，失去智能
- **预防措施**：健康检查在 decision 之前执行；LLM_RECOVERED 事件直接触发 RUNNING；测试覆盖
- **测试方式**：mock LLM 失败 5 次 → DEGRADED → mock 恢复 → 验证 10 轮内切回 RUNNING

### R5：LLM 输出 JSON 不合 schema

- **风险描述**：LLM 返回的 JSON 缺少字段、类型错误、或完全不是 JSON
- **触发条件**：prompt 变化、LLM 版本更新、极端输入
- **影响**：ContentGenerator 返回 None，本轮不回复
- **预防措施**：Pydantic schema 校验 + 容错解析（rfind 提取 JSON）；失败返回 None 不崩溃
- **测试方式**：构造 10 种畸形 JSON 输入 → 验证不崩溃、返回 None

### R6：DelaySendScheduler 重复发送

- **风险描述**：同一 action_id 被发送多次
- **触发条件**：schedule() 被调用两次、或 pre_send_check 后重入
- **影响**：bot 在群中刷屏，暴露 bot 身份
- **预防措施**：`mark_as_sending` 原子幂等（SQL WHERE status IN）；`has_pending` 去重检查
- **测试方式**：对同一 action_id 调用两次 schedule → 验证只发送一次

### R7：ShortTermMemory 消费竞态

- **风险描述**：consume_unread() 并发调用导致消息丢失或重复
- **触发条件**：多个线程同时调用 consume_unread
- **影响**：消息丢失或重复处理
- **预防措施**：原子交换 `unread = self._unread; self._unread = []` 在 lock 内完成；consume_unread 只由主循环调用
- **测试方式**：10 线程并发 consume_unread → 总消息数不变

### R8：memory_entry 过期策略误删重要记忆

- **风险描述**：expire_old 删除了 importance 高但 expires_at 过期的记忆
- **触发条件**：expires_at 设置不当（如所有记忆默认 7 天过期）
- **影响**：重要人物关系、核心事件丢失
- **预防措施**：importance ≥ 8 的记忆 expires_at=None（永不过期）；archive=True 时只标记 is_active=0 不删除
- **测试方式**：插入 importance=10 的记忆 → expire_old → 仍在数据库中

### R9：SQLite 写冲突

- **风险描述**：多线程同时写同一个 SQLite 数据库
- **触发条件**：ActionLog（DelaySendScheduler 线程写）和主循环线程同时操作
- **影响**：OperationalError: database is locked
- **预防措施**：每个 SQLite 连接用独立的 connection 对象；写操作用 `threading.Lock` 保护；WAL 模式
- **测试方式**：2 线程并发写 action_log 100 次 → 无异常
