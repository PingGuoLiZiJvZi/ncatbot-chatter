# 需求分析

## 核心目标

不是做一个功能性/指令型机器人，而是通过完备的系统设计，让 bot 在使用 QQ 时尽可能表现得像一个真人用户。

这意味着 bot 的每一个行为都需要经得起追问："如果是真人，他会这样做吗？"

### "真人感"量化验收指标

"真人感"本质上是**行为模式与人类用户统计特征的吻合度**。以下指标可以通过模拟运行或日志分析验证：

| 维度 | 指标 | 目标值 | 测量方式 |
|---|---|---|---|
| 发言频率 | 单日主动发言总条数 | ≤ 30 条（涵盖所有群+私聊） | 日志计数 |
| 发言分布 | 单群单小时内发言 | ≤ 5 条 | 日志计数 |
| 主动发言占比 | 主动发言 / 总发言 | ≤ 40%（即 ≥60% 为被动回复） | 日志统计 |
| 时间分布 | 0:00–7:00 时段发言数 | ≤ 月均总发言的 5% | 时段分布直方图 |
| 响应延迟 | 被动回复的延迟 | 5s–120s，服从长尾分布（不是固定间隔） | 延迟分布曲线 |
| 主动延迟 | 从"决定说话"到"实际发送" | 3s–300s，服从正态+均匀混合分布 | 延迟分布曲线 |
| 回复率 | 被 @ 后回复率 | ≥ 95% | 事件配对统计 |
| 话题连续性 | 回复与前文的语义相关度 | ≥ 0.5（embedding cosine），抽查 | 抽样评估 |
| 沉默率 | 收到消息但不回复的概率 | 非 @ 消息沉默率 ≥ 60% | 日志统计 |
| 同质化度 | 连续 5 条发言的句式/语气相似度 | LLM 判定"不相似" ≥ 80% | 定期抽样 |

> 以上指标在模拟阶段通过日志分析验证，上线后持续监控。目标值为初始基线，随迭代调整。

## 现状问题

### 架构层面

- **耦合严重**：`Chatter` 类承担了消息处理、插件路由、主动循环、响应解析、消息发送全部职责，没有清晰的分层
- **配置混乱**：项目自定义配置和 ncatbot 框架配置混在同一个 `config.yaml` 中，且新版 ncatbot 4.4.1 会对未知配置键报警告
- **无错误处理**：LLM 调用失败、JSON 解析失败时只是 print/空返回，没有任何降级或重试机制
- **无状态管理**：`is_character` 只是一个布尔值，没有 bot 状态机概念（未启动/运行中/暂停/错误恢复）
- **线程模型简陋**：主循环是一个 `while True` + `time.sleep`，停止时只能通过 `t.join()` 等待当前循环结束

### 记忆系统

- **只有两层**：短期（内存列表）+ 长期（SQLite 单表），缺少中层抽象（如"这件事是最近 10 分钟内发生的"）
- **长期记忆退化**：每次浓缩都是全量清空→写入一条新记录，旧记忆被不断覆盖丢失
- **无重要性权重**：所有消息平等对待，无法区分"今天要复习考试"和"今天天气不错"
- **无关系建模**：没有存储"我和谁更熟""谁经常和我聊天"这类关系信息
- **记忆检索被动**：LLM 每次都收到全部记忆，没有按相关性检索

### 行为表现

- **主动循环机械**：固定间隔检查→发送→等待，没有考虑"现在是凌晨 3 点，真人不会在这时说话"
- **回复频率无节制**：只靠随机间隔控制，没有"该说话才说，不该说就沉默"的判断
- **没有情绪/状态起伏**：每次都是同样的语气和态度，不会因为连续被骂而失落，也不会因为被夸而高兴
- **Prompt 简陋**：仅靠 `character.yaml` 的一段文字作为角色设定，没有行为准则、社交礼仪、语境感知的细粒度指导
- **JSON 格式约束脆弱**：LLM 常常在 JSON 外包裹额外文字，靠 `rfind('[')` 硬截取

### 功能缺失

- **无语境感知**：不知道正在进行的群聊话题、群成员之间的互动关系
- **无多模态**：无法处理图片、表情包、语音，只能处理文本
- **无主动行为**：除了发送消息，不能做任何其他事（换头像、改签名、发空间动态等，但这些可能超出需求范围）
- **无群聊差异化**：在不同群聊中表现完全由同一套 prompt 驱动，没有"每个群里的我都不一样"的感觉

## 目标架构原则

### 分层设计

```
配置层 → 基础设施层 → 记忆/状态层 → 决策层 → 表现层
```

- **配置层**：统一管理所有配置，与 ncatbot 框架配置完全分离
- **基础设施层**：封装 ncatbot API、LLM API、TTS 等外部依赖
- **记忆/状态层**：多层级记忆 + bot 内部状态管理
- **决策层**：根据当前状态 + 记忆上下文 + 角色设定，决定是否行动、如何行动
- **表现层**：将决策转化为具体的 QQ 消息/行为

### 关键设计目标

1. **可测试**：每一层都能独立测试，不依赖 ncatbot 框架运行
2. **可配置**：所有行为参数（频率、语气、记忆策略等）均由配置控制，不硬编码
3. **可观察**：有清晰的日志分级，能追踪 bot 的每一个决策依据
4. **可扩展**：新增功能（如处理图片、新的决策规则）不需要改动核心流程

## 功能需求

### 记忆系统（重构）

| 层级 | 存储 | 生命周期 | 说明 |
|---|---|---|---|
| 即时记忆 | 内存列表 | 单次循环 | 从上一次 LLM 调用之后收到的所有新消息 |
| 短期记忆 | 内存列表 | 最近 N 条/分钟 | 最近一段时间的对话上下文，LLM 直接可见 |
| 工作记忆 | 内存 dict | 当前会话 | 当前正在聊的话题、本轮发言的意图等 |
| 长期记忆 | SQLite（见下方表结构） | 永久 | 按重要性加权 + 按相关性检索的持久记忆 |

- 记忆需包含**重要性评分**，由 LLM 在浓缩时评定，而不是等量存储
- 长期记忆支持**语义检索**：不是把全部记忆塞给 LLM，而是只取相关的几条
- **关系图谱**：记录"谁是谁""谁经常和谁互动""我和某人的互动频率"

#### 长期记忆数据库表结构

```sql
-- 记忆条目表
CREATE TABLE memory_entry (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    chat_type   TEXT    NOT NULL,          -- 'private' | 'group'
    chat_id     TEXT    NOT NULL,          -- QQ号 或 群号
    timestamp   TEXT    NOT NULL,          -- ISO 8601: 2026-05-04T14:30:00
    importance  INTEGER NOT NULL DEFAULT 5, -- 1(琐碎) ~ 10(核心身份信息)
    summary     TEXT    NOT NULL,          -- 浓缩后的记忆文本（面向 LLM 检索）
    keywords    TEXT,                      -- 逗号分隔的关键词，辅助全文检索
    embedding   BLOB,                      -- 可选：文本 embedding bytes（用于语义检索，MVP 可暂缓）
    created_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);

-- 关系图谱表
CREATE TABLE relationship (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_a    TEXT    NOT NULL,          -- 实体标识，如 'user_3040802074' 或 'self'
    entity_b    TEXT    NOT NULL,          -- 实体标识
    rel_type    TEXT    NOT NULL,          -- 'friendship' | 'acquaintance' | 'enemy' | 'stranger'
    intimacy    INTEGER NOT NULL DEFAULT 0, -- -100(敌对) ~ 0(陌生人) ~ 100(挚友)
    note        TEXT,                      -- 关系备注："一起去过漫展""经常借钱不还"
    updated_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);

-- 实体档案表（对人/对群的认知）
CREATE TABLE entity_profile (
    entity_id   TEXT    PRIMARY KEY,       -- 'user_<QQ号>' | 'group_<群号>'
    entity_type TEXT    NOT NULL,          -- 'user' | 'group'
    display_name TEXT   NOT NULL,          -- 最新已知的昵称/群名
    summary     TEXT,                      -- LLM 维护的一段人物/群描述
    updated_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);
```

#### 记忆生命周期

```
消息到达 → 即时记忆
         ↓ (下次 LLM 调用时)
         短期记忆（最近 N 条，LLM 直接可见）
         ↓ (每轮对话结束后)
         检查：read_memory 条数 > threshold ?
         ├─ 是 → ConcentrateLLM 浓缩 → memory_entry (importance ≥ 阈值才入库)
         │                                    ↓
         │                              relationship / entity_profile 增量更新
         └─ 否 → 保留在 pending 中，等待下一轮合并
```

与旧版的关键区别：
- **不再全量清空**：长期记忆是追加式的，旧记忆不会因为一次浓缩就丢失
- **重要性过滤**：琐碎内容（"今天天气不错"）不入长期记忆，只保留有价值的
- **实体独立**：关系图和人物认知独立于消息记忆，不会被消息浓缩覆盖

### Bot 状态系统

- **活跃度曲线**：一天中不同时段应有不同的活跃倾向（深夜低、白天高）

#### 情绪模型（MVP 最小可行维度）

第一版只设 3 个维度，避免过度复杂：

| 维度 | 范围 | 含义 | 典型变化规则 |
|---|---|---|---|
| **valence**（愉悦度） | -1.0 ~ 1.0 | 正负向情绪：高兴↔不高兴 | 被夸 +0.1，被怼 -0.15，随时间向 0 回归 |
| **energy**（精力） | 0.0 ~ 1.0 | 当前精力水平：亢奋↔疲惫 | 每次发言 -0.05，每小时 +0.1 恢复，睡眠时段快速恢复 |
| **interest**（兴趣度） | 0.0 ~ 1.0 | 对当前话题/聊天的参与意愿 | 话题相关时 +0.2，无关时 -0.05，长时间沉默后 +0.1 |

三个维度共同影响：
- **是否发言**：`energy < 0.15` 时不主动发言（被动回复仍可）
- **发言语气**：`valence` 偏高时语气轻快，偏低时冷淡或消极
- **发言长度**：`energy + interest` 越高，回复越可能展开

为什么这 3 个就够了（MVP）：
- `valence` 覆盖"情绪好坏"→ 语气变化
- `energy` 覆盖"想说不想说"→ 发言频率
- `interest` 覆盖"对这个话题感不感兴趣"→ 话题跟随行为
- 三维已可组合出足够丰富的行为模式，后续可加 `anxiety`（焦虑/犹豫）、`playfulness`（玩心）等

- **社交能量**：类似"社交电池"，一段时间内说话太多会消耗，慢慢恢复

### 决策系统：DecisionEngine 完整流水线

DecisionEngine 是架构的核心。每次主循环触发时，按以下流水线执行：

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1: 状态更新                                       │
│  ├─ 更新情绪维度 (valence/energy/interest)               │
│  ├─ 更新社交能量 (social_battery)                        │
│  ├─ 更新活跃度权重 (基于当前时段)                         │
│  └─ 读取各 chat 的待处理消息 (unread → read 转移)         │
├─────────────────────────────────────────────────────────┤
│  Phase 2: 被动回复检查 (P0 — 必须执行)                    │
│  ├─ 遍历所有 chat 的即时记忆                              │
│  ├─ 规则判断：是否被 @？是否被私聊？是否有人问自己问题？    │
│  ├─ 命中 → 标记"必须回复"，加入响应队列                   │
│  └─ 此阶段不经过 LLM，纯规则判断                          │
├─────────────────────────────────────────────────────────┤
│  Phase 3: 主动发言评估 (P1 — 保守)                        │
│  ├─ 步骤 3.1: 硬性门槛检查                               │
│  │   ├─ energy < 0.15 → 本轮不主动发言                   │
│  │   ├─ social_battery < 0.2 → 本轮不主动发言            │
│  │   ├─ 当前时段权重 < 0.1 (深夜) → 不主动发言            │
│  │   ├─ 本轮已有被动回复 ≥ 3 条 → 不追加主动发言          │
│  │   └─ 任意命中 → 跳过 3.2，直接进入 Phase 4            │
│  ├─ 步骤 3.2: LLM 决策（仅当 3.1 全部通过）               │
│  │   ├─ 输入：短期记忆 + 长期记忆检索结果 + 情绪状态       │
│  │   ├─ Prompt 强调："除非有强烈发言动机，否则保持沉默"    │
│  │   ├─ LLM 输出：是否发言 + 对谁说 + 简要意图            │
│  │   └─ 如决定发言 → 加入响应队列                         │
│  └─ 最多产生 1 条主动发言（防止一次循环刷屏）              │
├─────────────────────────────────────────────────────────┤
│  Phase 4: 内容生成 (仅响应队列中的条目)                    │
│  ├─ 对每条待回复的 chat，精选相关记忆                      │
│  ├─ 组装 prompt：角色设定 + 行为准则 + 情绪状态 + 记忆     │
│  ├─ LLM 生成具体回复文本                                  │
│  └─ 输出：消息文本 + 目标 chat + 可选的 @/引用             │
├─────────────────────────────────────────────────────────┤
│  Phase 5: 后处理                                         │
│  ├─ 再次检查：该消息发出后是否合理？（紧急刹车）            │
│  │   └─ 规则：单群最近 3 条都是自己的发言 → 拦截          │
│  ├─ 随机延迟 (见主动发言保守设计)                          │
│  ├─ 通过表现层发送                                        │
│  └─ 更新状态：消耗 energy/social_battery，记录日志         │
└─────────────────────────────────────────────────────────┘
```

关键设计原则：
- **Phase 3.1 硬性门槛是纯规则**，不消耗 LLM token，保证基本安全
- **Phase 3.2 的 LLM prompt 偏见是"不说话"**：只有明确找到说话理由才会发言
- **Phase 5 紧急刹车**：即使 LLM 决定说话，规则层仍可最后拦截（如短时间内说过多）

### 消息处理与主动发言保守设计

#### 被动回复（高优先级，不受限制）

- 被 @ 的消息 → 必须回复（≥95% 回复率）
- 私聊消息 → 必须回复（≥95% 回复率）
- 群聊中明确提到自己名字/昵称 → 高概率回复（≥80%）
- 被动回复不受 energy/social_battery 硬性门槛限制（但语气会受影响）

#### 主动发言（保守设计，多层压制）

默认假设是"不说话"，需要证据支持才发言。压制层级从严格到宽松：

| 压制层 | 条件 | 效果 |
|---|---|---|
| 第一层：时段压制 | 当前时间在 0:00–7:00 | 一律不主动发言（被动回复照常） |
| 第二层：能量压制 | `energy < 0.15` 或 `social_battery < 0.2` | 本轮不主动发言 |
| 第三层：频率压制 | 本轮已有 ≥ 1 条主动发言 | 本轮不再追加 |
| 第四层：密集压制 | 该群最近 10 分钟内自己已发言 ≥ 3 条 | 不在该群主动发言 |
| 第五层：独白压制 | 该群最近 5 条消息中 ≥ 3 条是自己发的 | 不在该群主动发言（防止刷屏独白） |
| 第六层：LLM 判断 | 前五层全部通过后，由 LLM 判断 | Prompt 明确要求列出"为什么要说话的理由" |

#### 发言响应延迟策略

不使用固定延迟，而是分层随机：

- 被动回复延迟：正态分布 N(30s, 15s)，截断到 [5s, 120s]
- 主动发言延迟：正态分布 N(120s, 60s)，截断到 [30s, 300s]
- 同群连续发送最小间隔：120s（两条消息之间至少隔 2 分钟）
- 延迟目的：模拟真人"看到→思考→打字→发送"的过程，同时避免瞬时回复暴露 bot 身份

### 角色系统

- 角色设定不再是单段文本，而是结构化的：性格维度、说话风格、知识背景、人际关系、禁忌话题等
- 支持**角色一致性检查**：LLM 生成回复后，自查是否符合角色设定（可选）
- Prompt 中区分**角色描述**（我是谁）和**行为准则**（我怎么做），以及**上下文格式化**

## 非功能需求

### 可靠性

- LLM 调用失败时自动重试（指数退避），多次失败后降级为"暂时沉默"
- 消息发送失败时记录原因，不丢弃状态
- 主循环异常不导致 bot 退出，仅记录并等待下一轮

### 性能

- 长期记忆检索应控制在毫秒级（SQLite 满足需求，除非记忆量极大）
- LLM 调用的 context 应尽量精简，通过记忆检索减少 token 消耗
- 不应在一次循环中阻塞过久（单次 LLM 调用时间 < 30s）

### 安全性

- API key 不硬编码，不从 yaml 明文读取（考虑环境变量或加密存储）
- 防止 prompt injection：用户消息中可能包含试图覆盖 system prompt 的内容
- 防止 bot 被诱导做出不符合角色的行为（如被套出系统提示词）

## 迁移计划与阶段验收标准

### Phase 1: 基础设施搭建（预计 1–2 天）

| 任务 | 内容 |
|---|---|
| 配置分离 | 创建 `conf/bot.yaml`、`conf/character.yaml`、`conf/prompt.yaml`，与 ncatbot 配置完全独立 |
| LLM 封装 | `infra/llm_client.py`：统一 LLM 调用接口，含重试、超时、错误类型 |
| 日志系统 | 基于 Python logging，三级输出：控制台(INFO) + 文件(DEBUG) + 错误文件(ERROR) |
| ncatbot 适配 | `infra/bot_adapter.py`：封装 ncatbot 4.4.1 的注册/发送 API |

**Phase 1 验收标准：**
- [ ] 运行 `python main.py` 不报错，无 ncatbot 配置警告
- [ ] LLM 封装可独立调用并返回有效响应（提供独立测试脚本）
- [ ] 日志同时出现在控制台和文件中
- [ ] 配置文件全部用 .gitignore 排除，只提交 `.template` 版本

| 任务 | 内容 |
|---|---|

| 三层记忆 | 实现即时记忆→短期记忆→长期记忆的完整生命周期 |

**Phase 2 验收标准：**
- [ ] 模拟 100 条消息注入，验证记忆生命周期正常流转
- [ ] 长期记忆不会因浓缩而丢失旧数据（检查条目数 ≥ 浓缩前的有效条目数）
- [ ] 记忆检索返回与查询最相关的前 5 条（人工判断相关性）
- [ ] 关系图谱正确记录"谁在哪个群说了什么"（人工抽查 5 个 case）

### Phase 3: 情绪与状态系统（预计 1–2 天）

**Phase 3 验收标准：**

- [ ] 模拟连续被怼 20 条后，valence 明显下降（≤ -0.5）
- [ ] 模拟 0:00–7:00 时段，活跃度权重 ≤ 0.1
- [ ] energy 从 0.2 恢复到 0.8 耗时约 6 小时（不精确，在合理范围内即可）
